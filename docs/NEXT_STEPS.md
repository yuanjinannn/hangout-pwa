# 下一阶段工作清单

当前状态：PWA 原型已能跑通“地点搜索、导入地点、铸造活动票、投票、确认方案、举报审核、基础离线缓存”的主流程。

## 近期目标

先把它整理成一个稳定可演示版本，再决定是否进入真实后端和部署。

1. 演示稳定性
   - 固定一条 3 分钟演示路径：首页 -> 卡库搜地图 -> 铸造公开活动 -> 投票 -> 封存方案。
   - 保留高德 Key 的本地演示能力；如果要公开仓库或部署公网，再迁移到环境变量或服务端密钥配置。
   - 检查移动端小屏、桌面居中视图、刷新后本地状态恢复。

2. 产品收口
   - 明确核心对象：用户、地点、活动、参与人、投票、举报。
   - 决定活动是否继续“不区分类型”，还是加入吃饭/咖啡/夜景/桌游等活动类型。
   - 确认私密邀请链接是否需要过期、撤回、只读预览。

3. 技术整理
   - 拆分 `app.js`：状态、地点推荐、活动流程、渲染、事件绑定分开。
   - 把 mock 数据单独放入 `data/` 或 `fixtures/`。
   - 为核心纯函数补单元测试：地点评分、最佳时间、邀请链接编码/解码、举报阈值。

4. 后端化候选方案
   - 轻量方案：继续静态 PWA + localStorage，适合演示。
   - 中间方案：Node/Express + JSON 文件或 SQLite，适合小范围内测。
   - 正式方案：用户登录、活动表、地点表、投票表、举报表、邀请 token 表。

5. 部署前检查
   - 移除或保护任何不该公开的 API Key。
   - 准备应用图标、manifest、缓存版本号策略。
   - 确定地图服务调用限额和错误兜底文案。
   - 做一次真机安装测试。

## 建议的下一张票

优先做“演示稳定版 v0.1”：

- 创建固定演示数据重置入口。
- 加一个轻量测试页面或测试脚本，覆盖主流程。
- 修正文案里偏内部化的英文标签，让对外演示更自然。
- 保持当前视觉风格，只处理明显可读性和遮挡问题。
